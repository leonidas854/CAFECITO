# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

CaféSim — a 2D side-scrolling cafeteria queueing-theory simulator built in **Unity 6 (6000.3.16f1, URP, 2D)**. Academic project for 7th-semester *Simulación* + *Ingeniería de Software*. The simulation compares a physical-only ordering system against a hybrid physical+web system and is validated against an equivalent model in **ProModel**.

The repo is the Unity project root (no separate src/ — `Assets/`, `Packages/`, `ProjectSettings/` are siblings of this file). The C# solution file is `CAFECITO.slnx` (Visual Studio solution XML); `.csproj`/`.sln` files are generated by Unity and gitignored. There is no CLI build/test loop — open the project in Unity Hub with the exact editor version above. Tests use `com.unity.test-framework` and run from **Window → General → Test Runner** inside the editor.

At time of writing, `Assets/_Project/Scripts/{Core,Entities,UI,Events,Data}/` exist but are empty — the architecture is a planned skeleton, not yet implemented. `Assets/Scenes/SampleScene.unity` is the Unity default; the real scene `MainSimulation.unity` will live under `Assets/_Project/Scenes/`.

## Architecture (Clean Architecture, layered)

All gameplay code goes under `Assets/_Project/Scripts/`. The layer boundaries are load-bearing — do not let dependencies cross them in the wrong direction:

- **Core/** — Pure C# domain. LCG pseudorandom generator, exponential inter-arrival/service-time draws, queues, queueing-theory formulas (ρ, Lq, Wq, W). **No `MonoBehaviour`, no `UnityEngine` dependencies** so it stays unit-testable in isolation.
- **Entities/** — Unity-side view of agents (Cliente, Cajero, Barista). MonoBehaviours, state machines, DOTween-driven movement. Consumes Core; never the reverse.
- **UI/** — Dashboard, sliders for λ/μ, `Time.timeScale` controls, live charts. Reads/writes through Events, not direct references to Entities.
- **Events/** — Decoupling layer (C# events / Observer). When you need to wire two layers together, route the signal through here rather than holding a direct reference.
- **Data/** — `ScriptableObject` assets holding simulation parameters so scenarios can be swapped without code edits.

Other folders under `Assets/_Project/`: `Prefabs/`, `Scenes/`, `Art/`. The default Unity-generated assets (`InputSystem_Actions.inputactions`, `UniversalRenderPipelineGlobalSettings.asset`, default `Settings/`) live at `Assets/` root and should be left alone unless changing render pipeline / input config intentionally.

## Team rules that affect what you may change

These come from the README and are stricter than typical Unity hygiene — respect them:

- **`MainSimulation.unity` is owned by Integrante 3.** If your change would normally edit that scene, instead produce a Prefab and stop — do not modify the scene file.
- **Never commit directly to `main` or `develop`.** Branch naming is `feature/I[N]-<name>` or `bugfix/I[N]-<name>` where N is the team-member number. PRs target `develop`; `main` is release-only.
- **Always stage the `.meta` alongside the file it describes.** Missing `.meta` files silently break references on teammates' machines. This applies to every new file *and* every new folder.
- **No absolute paths in code.** Use `Application.dataPath` or `StreamingAssets`.

## Commits

Conventional Commits, **in Spanish, lowercase**: `tipo(alcance): descripción`. Allowed types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`. One atomic change per commit.

Examples from the README: `feat(core): agregar lcg con semilla fija`, `fix(ui): corregir slider de lambda que no actualizaba el manager`.

## Cross-platform gotcha

The team works across Arch Linux and Windows. `.gitattributes` forces LF on all text and Unity YAML files (`*.unity`, `*.prefab`, `*.asset`, `*.meta`, `*.mat`) and registers `unityyamlmerge` as the merge driver for them. Do not regenerate or rewrite these files with CRLF endings, and do not hand-edit Unity YAML conflicts without `unityyamlmerge` — it will corrupt scene/prefab references.

## Discrepancies to be aware of

- README lists **DOTween** as a dependency, but it is **not** in `Packages/manifest.json` yet — it must be added via the Asset Store / OpenUPM before any DOTween-using code will compile.
- README's "Guía de Inicio Rápido" code blocks contain copy-paste artifacts (stray ` ```cmd `, ` ```Text ` lines). Follow the intent, not the literal markdown.
