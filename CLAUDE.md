# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

CaféSim — a 2D side-scrolling cafeteria queueing-theory simulator built in **Unity 6 (6000.3.16f1, URP, 2D)**. Academic project for 7th-semester *Simulación* + *Ingeniería de Software*. The simulation compares a physical-only ordering system against a hybrid physical+web system and is validated against an equivalent model in **ProModel**.

Target platforms: Linux, Windows, WebGL.

The repo is the Unity project root (no separate src/ — `Assets/`, `Packages/`, `ProjectSettings/` are siblings of this file). The C# solution file is `CAFECITO.slnx`; `.csproj`/`.sln` files are generated by Unity and gitignored. There is no CLI build/test loop — open the project in Unity Hub with the exact editor version above. Tests use `com.unity.test-framework` and run from **Window → General → Test Runner** inside the editor (EditMode tab).

## Architecture (Clean Architecture, layered)

All gameplay code lives under `Assets/_Project/Scripts/`. Layer boundaries are load-bearing — do not let dependencies cross them in the wrong direction. The rule "Core knows nothing about Unity" is enforced by discipline only (no `.asmdef` files yet) — `grep UnityEngine` over `Scripts/Core/` and `Scripts/Events/` must return zero hits.

- **Core/** — Pure C# domain. No `using UnityEngine`, no `MonoBehaviour`. Holds:
  - `LCGRandomGenerator` (Numerical Recipes LCG, deterministic via seed), `ExponentialDistribution` (inverse-transform sampling).
  - `SimulationManager` (singleton orchestrator; consumers call `Configure(params)` then `Tick(dt)` each frame), `SimulationParameters` (validated POCO), `OrderSystem` (physical vs web routing).
  - `CustomerData` / `CustomerState` enum / `ProductType` enum / `ProductCatalog` (weighted product mix + per-product service times).
  - `Core/Queue/` — `IQueueDiscipline<T>`, `FifoQueue<T>`, `QueueController<T>` (wrapper that integrates queue length over time for empirical Lq).
  - `Core/Tables/` — `Table`, `TableManager`.
  - `Core/Metrics/` — `MetricCalculator` (analytical M/M/1 and M/M/c Erlang-C, plus empirical averages over finished customers), `MetricSnapshot` (immutable readout).
- **Entities/** — Unity-side view of agents. MonoBehaviours. Consumes Core; never the reverse.
  - `SimulationBootstrap` is the **single MonoBehaviour driver**: it instantiates a `SimulationConfig`/`SceneLayout` (creates defaults at runtime if none assigned), builds all placeholder furniture programmatically, calls `SimulationManager.Configure` in `Start`, and `Tick(Time.deltaTime)` in `Update`. There is no need to drag prefabs into the scene.
  - `CustomerSpawner` subscribes to `GameEvents.OnCustomerArrived/StateChanged/Left/SimulationReset` and creates / moves / destroys `CustomerEntity` GameObjects.
  - `CustomerEntity`, `ServerEntity` (cashier/barista/multi-skill), `TableEntity`.
  - `Entities/Movement/WaypointMover` — simple `Vector3.MoveTowards` with arrival threshold; DOTween-ready slot (DOTween itself is not yet in `Packages/manifest.json`).
  - `Entities/Layout/SceneLayout` — `ScriptableObject` with all positions (entry, exit, queue heads, table grid, etc.).
  - `Entities/Placeholders/PlaceholderShapes` — procedurally generated `Square` / `Circle` / `RoundedRect` sprites tinted via `SpriteRenderer.color`. Clients use `Circle`, servers/markers/tables use `RoundedRect`, seats use `Circle`. Swapping in real art = replace the sprite assignment in the Entity; no logic changes.
- **Events/** — Static `GameEvents` bus (Observer pattern). The only allowed wiring between Core and Entities/UI. **Every subscriber that registers in `OnEnable` must unregister in `OnDisable`** or events leak across scene reloads.
- **Data/** — `Scripts/Data/SimulationConfig.cs` is the `ScriptableObject` whose inspector exposes rates in clients/min (ergonomic) and converts to clients/s for the Core. The actual `.asset` instances live in `Assets/_Project/Data/`:
  - `SimulationConfig_Fisico.asset` — `webOrderProbability = 0` (baseline scenario).
  - `SimulationConfig_Hibrido.asset` — `webOrderProbability = 0.4`, λ slightly higher (the comparison scenario).
- **UI/** — Runtime dashboard. Built procedurally (no prefabs) by `DashboardUI` and instantiated automatically from `SimulationBootstrap.BuildDashboard()` when `buildDashboard = true`. Three regions:
  - `MetricsPanel` (top-left) — subscribes to `GameEvents.OnMetricsUpdated` and shows arrivals/serves/abandons/rejects, queue lengths, Lq, Wq, W, ρ_caja, ρ_barista, abandonment rate. Utilization rows colorize green/yellow/red at thresholds 0.7 / 0.9.
  - `ParameterControls` (top-right, in a ScrollRect) — sliders for λ, μ_caja, μ_barista fallback, paciencia, tiempo en mesa, p_web (applied hot via `SimulationManager.UpdateParameters`), plus cajeros/baristas/mesas/sillas/multi-skill (mesa/sillas/seed only take effect on Reset because `TableManager` is rebuilt there).
  - `SimulationControlBar` (bottom) — Pause/Resume, Reset, and 0.5x/1x/2x/5x/10x speed selector (modifies `Time.timeScale`).
  - `UIFactory` — central helpers (canvas, panel, text, button, toggle, labeled slider). Uses `Resources.GetBuiltinResource<Font>("LegacyRuntime.ttf")` for the font, no TextMeshPro dependency.
  - `EventSystem` is auto-created with the new Input System's `InputSystemUIInputModule` when present (the project has `activeInputHandler: 1` so the legacy module would not deliver clicks).

## Tests

`Assets/_Project/Tests/Editor/` holds NUnit suites that run via Test Runner (EditMode). Coverage as of the last check:

- Unit: `LCGRandomGeneratorTests`, `ExponentialDistributionTests`, `FifoQueueTests`, `QueueControllerTests`, `OrderSystemTests`, `MetricCalculatorTests`, `ProductCatalogTests`, `TableManagerTests`, `CustomerDataTests`, `SimulationParametersTests`.
- End-to-end (state-machine level): `SimulationManagerTests` — reproducibility under fixed seed, hard-limit enforcement, abandonment, multi-skill comparison, standing consumption.
- Integration vs theory: `SimulationIntegrationTests` — runs the full simulator for hours of simulated time and asserts that empirical ρ, Lq, Wq converge to analytical M/M/1 values, and that the hybrid scenario reduces cashier load vs the physical-only scenario.

When adding a new Core file, add its sibling test file. When changing `SimulationManager` flow logic, run `SimulationIntegrationTests` — those are the canaries.

## Team rules that affect what you may change

These come from the README and are stricter than typical Unity hygiene — respect them:

- **`MainSimulation.unity` is owned by Integrante 3.** If your change would normally edit that scene, instead produce a Prefab and stop. (Note: the scene does not exist yet — only `Assets/Scenes/SampleScene.unity` and the URP template are present.)
- **Never commit directly to `main` or `develop`.** Branch naming is `feature/I[N]-<name>` or `bugfix/I[N]-<name>` where N is the team-member number. PRs target `develop`; `main` is release-only.
- **Always stage the `.meta` alongside the file it describes.** Missing `.meta` files silently break references on teammates' machines. This applies to every new file *and* every new folder. The project follows Unity's pattern where new `.cs.meta` files contain only `fileFormatVersion: 2` + `guid:` — Unity fills the rest on import.
- **No absolute paths in code.** Use `Application.dataPath` or `Application.streamingAssetsPath`.

## Commits

Conventional Commits, **in Spanish, lowercase**: `tipo(alcance): descripción`. Allowed types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`. One atomic change per commit.

Examples from the README: `feat(core): agregar lcg con semilla fija`, `fix(ui): corregir slider de lambda que no actualizaba el manager`.

## Cross-platform gotcha

The team works across Arch Linux and Windows. `.gitattributes` forces LF on all text and Unity YAML files (`*.unity`, `*.prefab`, `*.asset`, `*.meta`, `*.mat`) and registers `unityyamlmerge` as the merge driver for them. Do not regenerate or rewrite these files with CRLF endings, and do not hand-edit Unity YAML conflicts without `unityyamlmerge` — it will corrupt scene/prefab references.

## Discrepancies to be aware of

- README lists **DOTween** as a dependency, but it is **not** in `Packages/manifest.json` yet — must be added via Asset Store / OpenUPM before any DOTween-using code will compile. Current movement uses `Vector3.MoveTowards` in `WaypointMover`.
- The `Tests/Editor/` folder works without an `.asmdef` because Unity's default `Assembly-CSharp-Editor` already references `Assembly-CSharp` and NUnit. Adding `.asmdef` files would let the Core run from `dotnet test` outside Unity (useful for ProModel cross-validation by I4) but is not required today.
- README's "Guía de Inicio Rápido" code blocks contain copy-paste artifacts (stray ` ```cmd `, ` ```Text ` lines). Follow the intent, not the literal markdown.
